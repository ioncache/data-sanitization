# Yarn Modern Migration

## Approach

Move the repository from Yarn Classic to the current stable Yarn release and use Yarn Plug'n'Play for dependency resolution. The migration updates package-manager metadata, regenerates the lockfile with Modern Yarn, removes `node_modules`-specific TypeScript configuration, and adjusts CI and development documentation to use Modern Yarn install commands.

## Pre-implementation

Create the `chore/yarn-modern-migration` branch before changing package-manager files.

## Steps

1. Update `package.json` to declare the Modern Yarn package manager and keep the Node version pinning aligned with the current Volta configuration.
2. Create `.yarnrc.yml` with `nodeLinker: pnp` and add package extensions for transitive peer metadata gaps surfaced by Yarn 4.
3. Regenerate `yarn.lock` with Modern Yarn so dependency resolution and lockfile format match the selected Yarn version.
4. Update `.github/workflows/ci.yml` to enable the project package manager and replace `yarn install --frozen-lockfile` with `yarn install --immutable`.
5. Update `.github/workflows/codeql-analysis.yml` to install dependencies with Yarn before CodeQL analysis so generated PnP runtime files exist on the runner.
6. Update `.gitignore` so local Yarn install metadata stays untracked.
7. Update `tsconfig.json` to avoid hard-coded `node_modules` type resolution.
8. Generate Yarn's VS Code SDK files so editor TypeScript resolution uses the PnP-aware TypeScript wrapper.
9. Update Oxlint, Oxfmt, and lint-staged configuration so generated Yarn SDK wrappers are not linted or formatted as source files.
10. Update `docs/development.md` with the new package-manager expectations and install command.
11. Run the package scripts that validate formatting, linting, building, and tests.

## Verification

Run `yarn --version`, `yarn install --immutable`, `yarn format:check`, `yarn lint --format=github`, `yarn build`, and `yarn test:coverage` after migration.

## Decisions

- Use the current stable Yarn 4 line rather than the `berry` alias so the package-manager version is explicit for contributors and CI.
- Use Plug'n'Play because the repository's formatting, linting, build, and test workflows pass after removing the `node_modules`-specific TypeScript `typeRoots` setting.
- Commit Yarn's generated VS Code SDKs because the TypeScript language service needs the PnP-aware wrapper to resolve package-backed configs and types in the editor.
- Filter generated SDK files in lint-staged because Oxlint exits with an error when every matched staged file is ignored.
- Track Yarn's editor SDKs but do not track cache or install state; Volta provides Yarn 4 locally, CI enables Corepack, and Yarn's `install-state.gz` is local cache state.
- Use `packageExtensions` for transitive peer metadata gaps so `yarn install --immutable` stays warning-free without adding direct project dependencies that the library does not use.
- Use `yarn install --immutable` in CI because Modern Yarn renamed the Classic `--frozen-lockfile` behavior.
- Install dependencies before CodeQL analysis because Plug'n'Play generates runtime metadata during install rather than relying on a committed `node_modules` tree.
